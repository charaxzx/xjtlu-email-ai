# XJTLU 邮件智能助手 · 演讲大纲

> 贴合本项目（Playwright + OWA + OpenAI 兼容 LLM + FastAPI Web），已排除团队合作维度。

---

## 1. 开场：一句话定位（约 20 秒）

- **我们做了什么**：面向西浦校园邮箱（OWA）的本地 Web 工具——用浏览器自动化登录你的收件箱，批量拉取**真实正文**，再用大模型生成**中文结构化摘要**。
- **给谁用**：每天被课程通知、活动、招聘、系统邮件淹没的同学；不想在网页里一封封点开、一条条记 deadline 的人。

---

## 2. 痛点与创新（对应「创新 / 解决真实问题」）

- **真实痛点**：学校邮箱走 OWA + 双因素认证，**没有开放 API**；学生只能网页里搜、点、滚，重要信息藏在正文里，列表预览不够用。
- **我们的思路**：不跟学校要接口——用 **Playwright 控制 Edge 打开真实 OWA**，在合法会话（Cookie / 交互式登录）下完成搜索、列表滚动、阅读窗格正文提取；再把文本交给 **LLM**，按你的指令输出摘要。
- **新意**：把「RPA 级别的邮箱操作」和「可换模型、可换指令的 AI 总结」绑成**一条本地流水线**，并针对 OWA 虚拟列表、阅读窗格懒加载等做了工程化处理（否则只有列表预览、正文截断，摘要不可信）。

---

## 3. 产品怎么用起来（对应「执行 / 可实际使用」）

按真实流程讲，建议演示顺序：

1. **本地启动**：`run_app.bat` / `python app.py`，浏览器打开 `http://localhost:8001`。
2. **账号与配置**：注册登录 → 设置里填 **AI 的 Base URL、Key、模型** → 连接邮箱：**粘贴 Cookie-Editor 导出的 JSON**，或使用 **交互式登录** 后「我已登录」写入会话。
3. **验证**：无头 **Cookie 检查** / 首页自动检查，避免白跑任务。
4. **日常模式**：关键词或留空看最新 → 选 **Prompt 预设**（综合摘要、求职、活动、Deadline 等）或自定义指令 → **生成摘要**；支持**多组关键词**各取若干封再合并去重。
5. **深度模式（亮点）**：先 **深度扫描**（大量列表 + 抽正文，落盘 `deep_scan_result.json`）→ 再按分类、日期、序号筛选 → **AI 分析**时从本地缓存读正文，**不再占用浏览器**，适合大批量、可重复分析。
6. **增值能力**：**定时摘要**（每日/每周）、侧栏历史、深度扫描下的分类统计与图表导出（若现场演示 ECharts 导出可提一句）。

**收束句**：从「连上邮箱」到「看到结构化中文摘要」是闭环，不是 Demo 截图。

## 4. AI 在哪里起作用（对应「AI 真正主导」）

分两层说：

### 产品里（用户感知的 AI）

- 对每封邮件：**并行**调用 LLM 做单封分析（批大小可配置），再 **一次汇总** 成最终输出；Prompt 里强调 **严禁编造**，与「必须基于提取到的正文」一致。
- 用户通过 **自然语言指令** 控制关注点（只要 deadline、只要活动、求职向等），配合内置 **Prompt 预设**，降低调 prompt 成本。

### 研发过程里（AI 如何帮你做成）

- OWA DOM 复杂、边界情况多：**AI 辅助写选择器逻辑、拆 bug、迭代提取与滚动策略**；最终代码里能看到针对列表/阅读窗格的专门处理，这不是「只贴了个 ChatGPT API」能糊弄过去的。

**时间紧时三句版**：抓取可信正文 + 双阶段 LLM（分封→汇总）+ 用户指令可配，且与 **深度扫描缓存** 架构一致。

---

## 5. 价值一句话 + 风险提示（对应「观众快速理解」）

- **价值**：把「在 OWA 里人工翻邮件」变成「**同一套会话下自动拉正文 + 一次读懂多封**」，特别适合期末、求职季、活动密集期。
- **诚实边界**：Cookie 会过期；OWA 改版可能要调选择器；数据在**本地**处理，提醒用户勿把密钥和 Cookie 提交到公开仓库（与 README 一致）。

---

## 6. 结尾（约 15 秒）

- 重复定位：**西浦邮箱场景下的「自动化阅读 + AI 摘要」本地助手**。
- 落点：**能连上、能抽全文、能按指令总结、能定时、能深度批量**——满足「能用」而不只是概念。
